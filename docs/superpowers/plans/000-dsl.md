# Arrow DSL for AI Agent Workflow Composition

> 這是一個**規劃工具**，不是執行引擎。
> DSL 幫助 AI agent 結構化思考、記錄、復現 workflow，展開後直接對應 tool use 指令。

## 問題

現在 AI agent 組合工具的方式有幾個根本問題：

1. **每次從零推理** — 對話結束後，tool use 的組合邏輯隨 context 消失，無法復現
2. **自然語言推理不可靠** — agent 用散文規劃多步操作，容易遺漏分支、難以 review
3. **JSON 佔太多 token** — 工具之間傳遞完整資料，LLM 花大量 token 在搬運而非決策
4. **人機無法圍繞同一份 data 工作** — 意圖在對話裡，資料在別處，產生大量例外

## 核心想法

用 Arrow-style DSL 作為 AI agent 的 **structured chain-of-thought**：

- agent 拿到任務時，先用 DSL 描述 pipeline 結構
- agent 自己把 DSL 展開成具體的 tool call 序列
- 成功的 pipeline 存為檔案，下次類似需求直接載入修改

**不需要 DSL interpreter / runtime。** DSL 是規劃語言，不是執行語言。Agent 本身就是 interpreter。

## 為什麼是 Arrow

Shell pipe (`|`) 只有線性單通道，表達力不足。Monad 太動態，pipeline 結構在執行前不可見。Arrow 卡在甜蜜點：

| Combinator | 語意 | 展開為 |
|------------|------|--------|
| `>>>` | 序列組合 | 依序 tool call |
| `***` | 並行組合 | 同一 message 內多個 tool call |
| `\|\|\|` | 分支/fallback | 第一個失敗才跑第二個 |
| `first` / `second` | 局部變換 | 只對 data 的一部分操作，其餘透傳 |
| `arr` | 純函數節點 | agent 內部的資料變換（不需要 tool call） |
| `loop` | feedback | retry / iterative refinement |

關鍵性質：**pipeline 結構在執行前完全可見。** 可以靜態驗證、可以讓人 review、可以存檔復現。

## 節點設計原則

DSL 的節點描述**目的**，不綁定特定 tool。每個節點包含：

- **purpose** — 這一步要達成什麼（必要）
- **reference tool** — 已知能完成此目的的 tool 名稱（選填，僅供參考）
- **input / output** — 這一步期待什麼、產出什麼的簡述

Agent 拿到 pipeline 時，根據 purpose 和自己可用的 tool 來決定怎麼展開。Reference tool 只是提示，不是約束。不同環境的 agent 可以用完全不同的 tool 來實現同一條 pipeline。

這讓 pipeline 成為**跨 agent 的 workflow 描述**，而不是某個特定 agent 的操作腳本。

## DSL 範例

### 基本 pipeline

```
read(source: "data.csv")                     -- 讀取資料來源
                                              -- ref: Read, cat, fetch
  >>> parse(format: csv)                      -- 結構化原始資料
  >>> filter(condition: "age > 18")           -- 篩選符合條件的記錄
  >>> (count *** collect(fields: [email]))    -- 並行：統計數量 & 收集欄位
  >>> format(as: report)                      -- 組織成人類可讀的輸出
```

### 帶 fallback

```
(fetch(url: endpoint)                         -- 從遠端取得資料
                                              -- ref: WebFetch, curl
  ||| load(from: cache, key: k))              -- 遠端失敗時讀快取
  >>> transform(mapping: schema_v2)           -- 轉換成目標格式
  >>> write(dest: "output.json")              -- 寫入結果
                                              -- ref: Write, fs.writeFile
```

### 帶 feedback loop

```
loop (
  generate(artifact: code, from: spec)        -- 根據規格產生程式碼
                                              -- ref: Write, Edit
    >>> verify(method: test_suite)             -- 執行驗證
                                              -- ref: Bash("npm test"), pytest
    >>> evaluate(criteria: all_pass)           -- 判斷是否通過
)
```

### 節點不綁 tool 的好處

同一條 pipeline，不同 agent 的展開方式不同：

```
read(source: "data.csv")

  Claude Code agent  →  Read tool
  shell agent        →  cat data.csv
  browser agent      →  fetch("/api/data.csv")
  人類               →  打開 Excel
```

## 組成元件

### 1. Skill：教 agent DSL 思維

核心內容：

- Arrow combinator 的語意和使用時機
- 從使用者意圖推導 pipeline 結構的思路
- 展開規則：每個 combinator 怎麼對應到 tool call
- 已有 pipeline 的複用和修改策略

Skill 讓 agent 在「規劃」和「執行」之間有一個明確的中間步驟。

### 2. Parser / Checker：純結構檢查

用 **OCaml** 實作。編譯為單一 binary，只做 parse 和結構驗證，不做語意檢查，不執行任何事。

選 OCaml 的理由：ADT + pattern matching 天生適合 AST 處理、`ocamlopt` 編譯出原生執行檔不需要 runtime、parser 工具鏈成熟（DSL 夠小，手寫 recursive descent 即可）。

#### DSL 語法元素

| 語法 | 意義 |
|------|------|
| `name(key: value, ...)` | 節點宣告，至少要有 purpose 描述 |
| `>>>` | 序列組合 |
| `***` | 並行組合 |
| `\|\|\|` | 分支 / fallback |
| `( )` | 群組 |
| `loop ( )` | feedback loop |
| `-- 文字` | 註解（purpose 描述、reference tool 等） |

#### 檢查規則

只檢查結構：

- 括號配對
- `***` 的兩端最終要有合併點
- `|||` 的兩條路徑都要有輸出
- `loop` 內要有判斷節點（作為終止條件）
- 每個節點至少要有 purpose（節點名稱或註解）

不檢查：

- 語意相容性（agent 自己判斷）
- reference tool 是否存在（那是 agent 的事）
- tool 的具體參數格式（agent 展開時處理）
- 任何需要執行才能知道的事

#### 輸出

Parser 產出 AST，Checker 回傳結構性錯誤列表。可以包成 CLI 或 MCP tool。

目的是在 agent 展開 pipeline 之前，擋住明顯的結構錯誤。

### 3. Pipeline 儲存：累積式工作

成功的 pipeline 存為檔案（markdown / 專用格式），包含：

- DSL 本體
- 使用場景的描述
- 各節點的 purpose 和 reference tool（供其他 agent 參考）
- 上次執行的結果摘要

下次類似需求時，agent 載入已有的 pipeline，局部修改而非從頭推理。

## 不做的事

- **不做 DSL interpreter** — agent 自己展開 DSL，不需要額外的執行引擎
- **不做新的 tool 系統** — 直接複用現有的 tool use 機制
- **不做視覺化編輯器**（初期）— 人要看 pipeline 用 json-render 或 markdown 就夠
- **不做型別推導** — 初期靠 agent 自己判斷型別相容性，Checker 可以後加

## 預期效果

| 現在 | 之後 |
|------|------|
| 每次對話從零推理工具組合 | 載入已有 pipeline，局部修改 |
| 自然語言規劃，難以 review | DSL 結構一目了然 |
| 對話結束，組合邏輯消失 | pipeline 存檔，可復現 |
| 工具間全量傳資料燒 token | DSL 層只傳 reference / schema |
| 人跟 AI 用不同介面理解同一件事 | DSL 是共同語言 |

4. 建立 pipeline 儲存的慣例（位置、格式、命名）
